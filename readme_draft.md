# 雙式記帳 CLI 系統

## Proposal Report

### 動機與目標  
許多自由工作者、小型商家或個人需要一套**可稽核、符合會計原則**的記帳工具，但市面上的試算表或手機 App 多為單式記帳，無法確保借貸平衡。  

本專案的目標是開發一個**命令列雙式記帳系統**，同時作為資料結構效能的實驗平台。使用者能快速記錄交易、使用巨集減少重複輸入，並透過**匯入／匯出**、**自動儲存**確保資料安全。額外實作**復原／重做**、**週期性交易**與**預算追蹤**，使工具具備基礎的個人財務管理能力。

### 預期功能  
- ✅ 新增交易（借方、貸方、金額、說明、註解）  
- ✅ 巨集（午餐、公車、雜支）  
- ✅ 查看交易列表（依日期或金額排序）  
- ✅ 帳戶餘額報表  
- ✅ 刪除交易（自動重新編號）  
- ✅ 編輯交易（修改任何欄位）  
- ✅ 加入／編輯註解  
- ✅ 復原／重做（最多 20 層）  
- ✅ 週期性交易（建立規則、到期自動新增）  
- ✅ 預算追蹤（每月預算 vs 實際支出）  
- ✅ 搜尋交易（關鍵字、金額區間、帳戶）  
- ✅ 匯出／匯入 JSON / CSV（合併或取代）  
- ✅ 效能測試（比較 list 與 dict 操作速度）  
- ✅ 自動儲存（每次變更即寫入檔案）  
- ✅ 事件迴圈命令列介面（help, exit）

### 使用技術  
- **語言**：Python 3  
- **資料儲存**：JSON 檔案（交易日誌、週期規則、預算設定分開儲存）  
- **命令列介面**：原生 `input()` 事件迴圈  
- **表格顯示**：`rich` 套件（`rich.table`, `rich.panel`, `rich.console`）  
- **匯入／匯出**：Python 內建 `json` 與 `csv` 模組  
- **日期處理**：`datetime`, `timedelta`  
- **效能測試**：自訂 `perftest` 指令，比較 list (O(n)) vs dict (O(1) 平均)  
- **單元測試（選配）**：`unittest` 框架（可獨立執行）

## Prototype Report

### 目前進度  
已完成完整可運作的**事件迴圈雙式記帳 CLI**，核心功能如下：

- ✅ **交易管理**：`add`, `list`, `balance`, `delete`, `edit`, `comment`  
- ✅ **巨集系統**：內建 `lunch` (Food↔Cash)、`bus` (Transport↔EasyCard)、`misc` (Miscellaneous↔Cash)  
- ✅ **復原／重做**：每項修改操作前自動推入堆疊，支援 20 層歷史  
- ✅ **週期性交易**：  
  - `recur add` 建立規則（描述、金額、借貸帳戶、間隔天數）  
  - `recur list` 顯示所有規則及下次到期日  
  - `recur delete` 移除規則  
  - `recur run` 檢查並新增所有到期的交易，自動更新下次到期日  
- ✅ **預算追蹤**：  
  - `budget set` 為支出帳戶設定每月預算  
  - `budget show` 比較當月實際支出與預算（含總覽面板）  
- ✅ **搜尋**：關鍵字搜尋 (`search coffee`) 或互動式多條件篩選（金額、帳戶、關鍵字）  
- ✅ **匯入／匯出**：支援 JSON 與 CSV 格式，可合併或完全取代現有資料，保留註解欄位  
- ✅ **效能測試**：`perftest` 指令載入 10,000 筆測試資料，實際比較 list 與 dict 在隨機查詢及刪除後重建的執行時間  
- ✅ **自動儲存**：所有寫入操作（新增、刪除、編輯、匯入、復原/重做、週期執行）皆立即存檔，程式結束時再次儲存  
- ✅ **美觀報表**：使用 `rich` 顯示彩色表格，餘額正負以綠／紅標示，預算狀態以圖示顯示  
- ✅ **資料完整性**：刪除交易後自動重新編號 ID，匯入時處理型別轉換及缺漏欄位  

**程式碼結構**：單一檔案約 600 行，已分割為資料管理、核心功能、巨集、週期、預算、匯出入、效能測試與主迴圈，具備良好可讀性。

### 遇到的困難  

1. **CSV 匯入的型別問題**  
   - 問題：CSV 讀取時所有欄位皆為字串，導致金額與 ID 無法計算及排序。  
   - 解決：在 `import_transactions` 中明確將 `id` 轉為 `int`、`amount` 轉為 `float`，並確保 `comment` 欄位存在。

2. **重新編號與復原堆疊的互動**  
   - 問題：刪除交易後重新編號，若使用者復原，舊 ID 會與目前狀態衝突。  
   - 解決：復原／重做時直接覆蓋整個 transactions dict，不依賴 ID 連續性；ID 重整僅在刪除後立即執行，復原時會還原完整的 dict（含原來的 ID）。

3. **週期交易的「下次到期日」更新邏輯**  
   - 問題：若使用者手動修改交易日期，可能造成週期規則與實際記錄不一致。  
   - 解決：週期規則獨立儲存於 `recurring.json`，`recur run` 僅根據規則的 `next_due` 新增交易，不依賴已存在的交易記錄。使用者可手動編輯 JSON 調整 `next_due`。

4. **預算計算的帳戶範圍**  
   - 問題：所有借方帳戶都會被計入支出，但收入帳戶（如薪資）不應納入預算消耗。  
   - 解決：`budget show` 僅對已經設定預算的帳戶進行統計，未設定預算的借方帳戶忽略。使用者只需為「支出類」帳戶設定預算。

5. **大量測試資料產生與效能比較的再現性**  
   - 問題：測試資料需隨機但可重複，否則 `perftest` 結果難以比較。  
   - 解決：提供 `generate_test_data.py` 可產生固定隨機種子 (`random.seed(42)`) 的 10,000 筆交易，確保每次測試資料一致。
